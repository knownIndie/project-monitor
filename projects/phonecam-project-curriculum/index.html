GitHub-based skill assessment

I inspected your current GitHub profile↗ and excluded RealtimeChat.

Your actual baseline:

  • kontxt-cli↗ is your strongest engineering project. It has a published package, build tooling, deterministic behavior, and automated tests.
  • Luma↗ proves you understand Next.js, authentication, database modeling, payments, webhooks, and product workflows. It has no tests.
  • VoidYoutube↗ proves you can work with browser APIs, background service workers, DOM injection, and lifecycle events. It has a known multi-tab race condition and no tests.
  • Cloudline↗ is competent React work, but building another ordinary API-based frontend would add almost no value.
  • DevScreen↗ is still too early to count as strong evidence.
  • WeSplit↗ shows minimal SwiftUI exposure, but it is an archived tutorial-sized project. You should still be treated as a Swift beginner.
  • Your main weaknesses are native macOS development, automated testing outside kontxt-cli, concurrency, WebRTC, media buffers, and low-level Apple APIs.

These estimates assume:

  • One to two hours daily.
  • Mostly documentation-driven work.
  • You write the implementation yourself.
  • AI is used for explanations and debugging, not generating whole projects.
  • Every checkpoint is tested and documented before moving forward.

How to work without depending on AI

For every project:

  1. Read the listed documentation.
  2. Write a one-page explanation in your own words.
  3. Build the smallest successful experiment.
  4. Add tests.
  5. Intentionally trigger failures.
  6. Use the debugger and logs.
  7. Commit only after you can explain the implementation.
  8. Record the result in docs/learning-log.md.

Use AI only when:

  • Official documentation is ambiguous.
  • You have investigated the same error for at least 60 minutes.
  • You want your explanation reviewed.
  • You want test cases suggested after writing the implementation.

Do not ask AI to:

  • Generate the entire feature.
  • Design every type before you understand the API.
  • Rewrite working code you cannot explain.
  • Solve compiler errors before you read the error and relevant documentation.
